Story Map Format in Markdown

This file lives in the repo and IS the backlog. Which format feels right?

A

Nested Sections (Persona → Activity → Stickies)

Each persona is a heading. Activities are sub-headings. Stickies are checkboxes. Clean hierarchy, easy to scan, easy for the skill to parse the next unchecked item.

story-map.md
## Value Story
As a buyer I want to pay for goods so I can receive them.

## 🧑 Buyer
### Browse
- [x] View product listing
- [ ] Filter by category ← next
- [ ] Add item to cart

### Checkout
- [ ] Enter payment details
- [ ] Confirm order

## 🔧 Admin
### Orders
- [ ] View all orders
- [ ] Mark order fulfilled
B

Table (Left-to-right Activities × Personas)

Mirrors a physical story map board. Activities go left to right across the top. Personas are rows. Stickies live in the cells. Hardest for the skill to parse checkboxes from.

story-map.md
| Persona | Browse | Checkout | Post-purchase |
|---|---|---|---|
| 🧑 Buyer | [x] View listing<br>[ ] Filter | [ ] Payment<br>[ ] Confirm | [ ] Track order |
| 🔧 Admin | — | — | [ ] Fulfill order |
C

Flat Checklist with Tags

One flat list, every sticky is a checkbox. Persona and activity are inline tags. Simplest format, easiest for the skill to scan, least structured visually.

story-map.md
## Value Story
As a buyer I want to pay for goods so I can receive them.

## Stickies
- [x] [buyer/browse] View product listing
- [ ] [buyer/browse] Filter by category
- [ ] [buyer/browse] Add item to cart
- [ ] [buyer/checkout] Enter payment details
- [ ] [buyer/checkout] Confirm order
- [ ] [admin/orders] View all orders
- [ ] [admin/orders] Mark order fulfilled

My recommendation

A — Nested Sections. It reads like a real story map, the hierarchy is clear at a glance, and the skill can trivially find the next unchecked sticky by scanning top-to-bottom. Table format (B) looks like a board but is painful to maintain in markdown. Flat tags (C) lose the visual structure that makes a story map useful for review.
