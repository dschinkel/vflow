## <span style="color:#76a039">What This Repo Is</span>

A personal collection of custom Claude Code skills and supporting hooks. No application code. The goal is a library of `/commands` that any agent picks up automatically or a user invokes explicitly — covering the full development lifecycle from feature design through TDD, hexagonal architecture, and naming refactors.

---

## <span style="color:#76a039">What's Shipped</span>

### <span style="color:#76a039">`/refactor` — **complete and in use**</span>
The only fully implemented command. Proposes naming refactors using domain language, inside-out (constants → function names). Per-session it asks for a hypothesis and prediction, runs the rename cycle with accept/reject flow, produces a flat rename log and a color-coded Mermaid tree diagram, then writes a verdict/analysis/learnings block at the end. Total token count is auto-filled into the log at session end by the Stop hook.

**Files:**
- `commands/refactor.md` — canonical source, mirrored to `.claude/commands/refactor.md`
- `hooks/end-refactor-log-session-stats.sh` — Stop hook: fills `Total Tokens` in the log
- `hooks/start-refactor-skill.sh` — UserPromptSubmit hook: blocks `/refactor` if context window is too full (Sonnet: <90% remaining, Opus: <80% remaining)
- `hooks/tests/test-stop-hook-cleanup-trap.sh` — smoke tests for the Stop hook trap behavior
- `install.sh` — copies commands and hooks to `~/.claude/`, merges hook entries into `settings.json`

## <span style="color:#76a039">What's In Design / Not Yet Built</span>

### <span style="color:#76a039">`/feature` — **spec complete, not implemented**</span>
End-to-end feature development skill. Four phases: Drafter (guided questions → story map), Provocateur (challenges the draft), Persist (writes `story-map.md` + `story-map.html`), Implement (picks one sticky at a time, auto-invokes sub-skills). Story map is the living backlog; HTML board is always derived from the markdown. Decision Tree toggle is a future variant of the same data.
`docs/superpowers/specs/feature/2026-05-17-feature-skill-design.md`

### <span style="color:#76a039">`/tdd` — **spec complete today, not implemented**</span>
Full TDD session conductor. Invoked per sticky by the feature skill or standalone. Three opening questions set the mode: opt-in, outside-in vs inside-out, plan review (yes = human-in-the-loop with gates; no = Conductor mode, fully autonomous commits). Generates a `tdd-plan.md` before any code, then drives RED/GREEN/REFACTOR cycles. REFACTOR phase auto-invokes `/refactor`. Writes `tdd-implementation.md` incrementally. Rollback on intentional exit or unexpected termination — detected via `.tdd-context.json` state file.
`docs/superpowers/specs/2026-05-17-tdd-skill-design.md`

### <span style="color:#76a039">`/hexagonal` — **spec complete, not implemented**</span>
Two-skill pair. `hexagonal.md` enforces layered architecture rules (controllers → use cases → repositories → data) on any service-layer code, automatically during feature builds or explicitly via `/hexagonal`. `hexagonal-scaffold.md` creates the folder structure and a README with pseudocode when a new service is needed. Supports Node.js (JS Module Pattern) and Python (idiomatic functional). Optional style extraction from an existing reference codebase.
`docs/superpowers/specs/skills/2026-05-17-hexagonal-architecture-skill.md`

## <span style="color:#76a039">Key Conventions</span>

- `commands/` is canonical — always mirror changes to `.claude/commands/`
- `docs/refactorings/` is gitignored — session output, not committed
- When `/refactor` is invoked by `/tdd`, its logs go into the feature's TDD folder, not `docs/refactorings/`
- `/refactor` will need a small update to accept an output path override for this to work
- Brainstorming notes, random thoughts, and early research live in `.claude/brainstorming_custom-skills/`
